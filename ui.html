O que é SLA?

SLA (Service Level Agreement) é o tempo máximo esperado para que um erro seja respondido. O sistema monitora automaticamente o tempo decorrido desde o registro e sinaliza visualmente quando o prazo está próximo ou foi ultrapassado.

✅ OK — dentro do prazo
⚠️ ALERTA — próximo do limite (padrão: 15 min)
🚨 CRÍTICO — prazo ultrapassado (padrão: 30 min)

Os limiares de alerta e crítico são configuráveis em Configurações → Sistema → Limiares de SLA.

👤 Perfis de Usuário e Permissões

Todo usuário cadastrado tem um perfil base que define suas permissões padrão. Desde a v237 é possível também personalizar as permissões individualmente para cada usuário — sem mudar o perfil dele — através do editor de permissões em Configurações → Usuários.

🔑
ADMIN
Acesso total ao sistema. Pode gerenciar usuários, configurações, cadastros, executar diagnósticos, arquivamentos e importação de dados. Vê dados de todos os setores. É o único perfil que pode acessar a aba Auditoria.
📋
CONFERENTE
Visualiza e edita todas as solicitações e respostas de todos os setores. Acessa as Configurações para consulta. Não gerencia usuários nem executa arquivamento. Ideal para supervisores e líderes de equipe.
✅
RESPOSTA
Registra novas solicitações e responde erros, mas apenas dos setores atribuídos a ele. Não vê dados de outros setores. Não acessa as Configurações. Ideal para colaboradores responsáveis por setores específicos.
👁️
ESPECTADOR
Somente visualização: acessa o Dashboard e a aba Ajuda. Não pode registrar nem responder solicitações. Os formulários ficam desativados. Indicado para gestores que acompanham métricas sem interagir com os dados.
🔧 Como funciona o editor de permissões individuais (v237 · atualizado v241)

Cada usuário pode ter permissões personalizadas que sobrescrevem os padrões do perfil dele. Isso permite, por exemplo, dar acesso à aba Auditoria para um CONFERENTE específico, remover a ação de Excluir para um ADMIN, ou ocultar subabas do Dashboard que não são relevantes para determinado usuário.

Como editar:

  1. Vá em Configurações → Usuários
  2. Clique em Editar no card do usuário desejado
  3. Selecione o perfil base nos cards visuais — os checkboxes de permissão serão pré-preenchidos com os padrões daquele perfil
  4. Marque ou desmarque livremente as Abas acessíveis, Ações disponíveis e Subabas do dashboard
  5. Para voltar ao padrão do perfil, clique em "↺ Restaurar padrões do perfil"
  6. Clique em Salvar usuário

O editor possui 3 colunas: Abas acessíveis (quais abas do menu aparecem), Ações disponíveis (o que o usuário pode fazer) e Subabas do dashboard (quais das 4 subabas — Colaboradores, Solicitantes, Ranking, Erros — ficam visíveis). As preferências de subabas ficam salvas na planilha junto com as demais permissões.

No card de cada usuário cadastrado, os chips mostram a contagem real de permissões. Quando as permissões estão customizadas (diferentes do padrão do perfil), aparece o símbolo ✎ no chip.

⚠️ Segurança: As permissões do frontend controlam o que cada usuário vê e acessa. O backend ainda valida o perfil para operações críticas (exclusão, gerenciamento de usuários). Permissões customizadas não substituem as validações de servidor.

📖 Guia Detalhado de Uso

📝 Aba Solicitação — Registrar um erro

Use esta aba para registrar um novo erro encontrado em uma requisição farmacêutica. Apenas usuários com a ação Registrar solicitação habilitada têm acesso.

Passo a passo:

  1. Requisição — Informe o número da requisição onde o erro foi encontrado (ex: 123456). Máximo 30 caracteres.
  2. Solicitante — Selecione na lista a farmácia ou unidade que está solicitando a correção.
  3. Tipo de Erro — Escolha o tipo de erro da lista cadastrada. Cada erro tem uma classificação de gravidade: NA Leve Grave Gravíssimo.
  4. Detalhamento — Descreva o problema com o máximo de detalhes possível. Máximo 2000 caracteres. Um bom detalhamento facilita a correção rápida.
  5. Setor/Local — Selecione onde o erro ocorreu. Somente usuários com acesso ao setor poderão ver esta solicitação.
  6. Diferença no Valor? — Indique SIM ou NÃO. Se SIM, o Dashboard exibirá este registro no indicador de Dif. Valor R$.
  7. Confirmação Médica? — Indique se é necessária confirmação médica. Se SIM, a linha aparecerá destacada em lilás na aba Resposta, chamando atenção da equipe.
  8. Clique em "Enviar Solicitação". O sistema gera automaticamente um ID único (ex: SOL-20260410-001) e registra a data/hora do pedido.
💡 Adicionar mais erros à mesma requisição:
Uma mesma requisição pode ter múltiplos erros (sequências). Use a opção "Adicionar erro à requisição existente", informe o ID da solicitação já criada, e um novo erro (sequência 2, 3...) será vinculado à mesma requisição.
⚠️ Anti-duplicata: O botão de envio é bloqueado após o primeiro clique para evitar registros duplicados. Se o envio demorar, aguarde — não clique várias vezes.
✅ Aba Resposta — Responder e acompanhar erros

Esta é a aba principal do dia a dia. Exibe todas as solicitações abertas ou em correção do(s) setor(es) do usuário logado. Usuários RESPOSTA veem apenas os erros do seu setor; ADMIN e CONFERENTE veem tudo.

Como usar os filtros:

Requisição
Busca textual pelo número da requisição
Status
ABERTO, EM_CORRECAO, CORRIGIDO ou Todos
Solicitante

Setor
Filtra por setor/local específico

Mostra só erros com diferença de valor

Mostra só erros que requerem conf. médica

💾 Filtros persistentes: Os filtros que você aplica ficam salvos automaticamente em localStorage. Na próxima vez que abrir a aba Resposta, os mesmos filtros serão restaurados. O título "Solicitações do Setor" exibe um badge azul com a contagem de filtros ativos (ex: 3 filtros). Para limpar tudo, clique em "Limpar Filtros".

Registrar uma resposta:

  1. Clique em uma linha da tabela para selecioná-la — ela ficará destacada em azul
  2. O modal de resposta abrirá com os dados da solicitação selecionada
  3. Selecione o Responsável pelo Erro na lista de colaboradores
  4. Informe se Houve Diferença no Valor (SIM/NÃO). Se SIM, preencha o valor em R$
  5. Adicione Observações se necessário (máx. 1000 caracteres)
  6. Clique em "Salvar Resposta"

Editar uma resposta já registrada:

  1. Clique no botão ✏️ Editar na linha correspondente
  2. O modal de edição abrirá. Dependendo do seu perfil você poderá editar os dados da solicitação (ADMIN/CONFERENTE) ou apenas a resposta (RESPOSTA)
  3. Faça os ajustes e clique em Salvar

Indicadores visuais na tabela:

  • Linha com fundo amarelo claro + ⚡ — solicitação marcada como urgente
  • Linha com fundo lilás — requer confirmação médica
  • Linha com fundo verde claro — solicitação normal
  • Linha com fundo azul — linha selecionada pelo clique
  • ALERTA e CRÍTICO — indicadores de SLA na coluna Tempo

Excluir uma solicitação: o botão 🗑️ aparece apenas para quem tem a ação "Excluir solicitação" habilitada. A exclusão remove a solicitação e todos os seus erros/respostas vinculados e é irreversível.

📊 Aba Dashboard — Análise e Relatórios

Painel de análise com KPIs, gráficos e relatórios. Disponível para todos os perfis, mas usuários RESPOSTA veem apenas dados do seu setor.

Quatro sub-abas disponíveis:

👥 Dashboard Colaboradores
Analisa os erros por responsável. Identifica quem tem mais correções pendentes e o tempo médio de atendimento por colaborador.
🏪 Dashboard Solicitantes
Analisa erros por farmácia/solicitante. Mostra quais unidades geram mais erros e os tipos mais frequentes por solicitante.
🏆 Ranking
Classificação comparativa de colaboradores e solicitantes por volume de erros, tempo médio e taxa de atendimento no período.
⚠️ Erros
Análise por tipo e classificação de erro (NA / Leve / Grave / Gravíssimo). Identifica os tipos mais recorrentes no período.

Filtros do Dashboard:

  • Data Inicial / Final — filtra pelo período de criação das solicitações
  • Timeline — visualização rápida: 7 dias, 15 dias, 30 dias, 3 meses ou 1 ano
  • Tipo de Erro — filtra por tipo específico
  • Responsável / Solicitante — filtra por colaborador ou farmácia

KPIs exibidos:

  • Total de Erros / Corrigidos / Abertos / Críticos SLA
  • Taxa de Atendimento (%)
  • Dif. Valor R$ — quantidade e soma total das diferenças de valor
  • Conf. Médica — quantidade e percentual de erros que requerem confirmação
  • Tempo médio, mínimo e máximo de resolução
  • SLA: % dentro do prazo e % fora do prazo

Gráficos disponíveis:

  • Ranking de erros por colaborador (barras horizontais)
  • Tipos de erro mais frequentes (pizza)
  • Erros por classificação de gravidade: NA / Leve / Grave / Gravíssimo (rosca)
  • Timeline de erros ao longo do tempo (linha)
💡 Performance: Os dados do Dashboard são cacheados por 10 minutos no servidor. Se você acabou de registrar uma solicitação e ela não aparece no Dashboard, aguarde alguns instantes e recarregue clicando em Atualizar Dados.
🔍 Aba Auditoria — Rastreabilidade total (ADMIN)

A aba Auditoria exibe o histórico completo de todas as ações de escrita realizadas no sistema. Disponível apenas para usuários com a ação Visualizar auditoria habilitada (por padrão, somente ADMIN).

O que é registrado:

  • Criação, edição e exclusão de solicitações e respostas
  • Cadastro, atualização e remoção de usuários
  • Alterações nas configurações (tipos de erro, setores, solicitantes)
  • Arquivamentos de dados e importações PROD→DEV

Cada registro inclui: data/hora, e-mail do usuário, tabela afetada, campo alterado, valor anterior e novo valor.

Os logs de auditoria ficam gravados na aba Auditoria da planilha do Google Sheets, onde podem ser consultados e filtrados diretamente.

⚙️ Guia de Configurações

A aba Configurações tem quatro seções acessíveis pelo menu horizontal. A visibilidade de cada seção depende das permissões do usuário.

📂 Tipos de Erro, Setores e Solicitantes

Estas três seções permitem gerenciar os cadastros que alimentam os dropdowns do sistema.

Tipos de Erro (Erros_Cadastro):

  • Adicione um novo tipo: preencha o nome e clique em "+ Adicionar"
  • Cada tipo tem uma classificação de gravidade: NA, Leve, Grave, Gravíssimo. Clique no badge de classificação para editá-la.
  • A lista é ordenada automaticamente por gravidade (NA → Leve → Grave → Gravíssimo) e depois alfabeticamente dentro de cada grupo.
  • Para remover um tipo, clique no ícone 🗑️. Atenção: tipos já utilizados em solicitações históricas continuarão aparecendo nos registros antigos.

Setores/Local (Setores_Local):

  • Cadastre os setores onde erros podem ocorrer. Os usuários RESPOSTA precisam estar vinculados a pelo menos um setor para receber solicitações.
  • Para liberar um usuário RESPOSTA para todos os setores, edite o usuário e marque o campo Setores como "*" (asterisco).

Solicitantes (Solicitantes):

  • Lista das farmácias ou unidades que podem solicitar correções. Aparecem no dropdown "Solicitante" da aba Solicitação.
👥 Gerenciar Usuários — Cadastro e Permissões

Apenas administradores têm acesso a esta seção. Aqui você cadastra, edita e remove usuários do sistema, além de definir perfil, setores e permissões individuais.

Cadastrar um novo usuário:

  1. Informe o E-mail Google do usuário (ex: colaborador@neoformula.com.br)
  2. Selecione o Nome na lista de colaboradores
  3. Escolha o Perfil clicando em um dos quatro cards visuais (Admin, Conferente, Resposta, Espectador)
  4. Ao selecionar o perfil, o painel de permissões aparece com checkboxes. As permissões são pré-preenchidas com os padrões do perfil escolhido. Você pode personalizá-las livremente.
  5. Para perfis RESPOSTA e ESPECTADOR: selecione os Setores que este usuário poderá visualizar. Para ADMIN e CONFERENTE o campo é preenchido automaticamente com * (todos os setores).
  6. Clique em "Salvar usuário"

Cards de usuários cadastrados:

Cada usuário cadastrado aparece como um card na parte inferior da tela. Os chips coloridos mostram:

  • 🗂 6/6 abas — quantidade de abas acessíveis (verde = todas, amarelo = parcial)
  • ⚡ 4/8 ações ✎ — ações disponíveis. O símbolo ✎ indica que as permissões foram customizadas

⚠️ Importante sobre e-mails: O acesso ao sistema é controlado pelo e-mail Google do usuário. Se o colaborador acessar com um e-mail diferente do cadastrado, ele não terá permissões e será tratado como ESPECTADOR.

🖥️ Sistema — Diagnóstico, SLA, Backup e Importação

🩺 Diagnóstico do Sistema:

  • Executar Diagnóstico — verifica a integridade das abas da planilha, configurações e conectividade. Exibe um painel visual com status de cada componente.
  • Diagnosticar e Corrigir — além de diagnosticar, tenta corrigir automaticamente problemas encontrados (abas faltando, dados corrompidos etc.)

🔧 Utilitários:

  • Limpar Cache — invalida o cache do servidor. Use quando dados parecem desatualizados após uma alteração manual na planilha.
  • ID da Pasta de Backup — configure o ID do Google Drive para onde os arquivos de arquivamento serão salvos.

⏱️ Limiares de SLA:

  • Minutos para Alerta — tempo após o qual a solicitação entra em estado ALERTA (padrão: 15 min)
  • Minutos para Crítico — tempo após o qual entra em estado CRÍTICO (padrão: 30 min)
  • Clique em "Salvar SLA" para aplicar

📦 Arquivamento Mensal:

Move os dados do período selecionado (mês/ano) para a planilha de backup do Looker Studio, removendo-os do app. Use para manter a base de dados ágil e com boa performance.

  • Selecione o Mês e o Ano a arquivar
  • Clique em "📤 Arquivar Mês Selecionado"
  • O sistema pede confirmação antes de executar. A ação não pode ser desfeita

📥 Importar dados da PROD (visível apenas no DEV):

Permite copiar dados da planilha de produção para o ambiente de desenvolvimento, facilitando testes com dados reais.

  • Selecione as tabelas que deseja importar usando os checkboxes
  • Use "Selecionar tudo" ou "Limpar seleção" para gerenciar rapidamente
  • Clique em "📥 Importar selecionados"
  • O sistema pede confirmação mostrando exatamente quais tabelas serão sobrescritas
  • Após a importação, o resultado exibe quantas linhas foram importadas por tabela
  • ⚠️ Os dados do DEV são completamente substituídos nas tabelas selecionadas

🔌 Referência Técnica — API Backend

O sistema usa Google Apps Script (V8) como backend. O frontend chama as funções via google.script.run. Todas retornam { ok: boolean, errors: string[], debugId: string, ...dados }.

⚡ Arquitetura e Performance

Stack tecnológica:

  • Backend: Google Apps Script V8 — arquivos .gs organizados por responsabilidade
  • Frontend: SPA vanilla (HTML + CSS + JS inline, sem frameworks, sem npm)
  • Banco de dados: Google Sheets (abas = tabelas, colunas por índice numérico)
  • Deploy: Google Web App via CLASP — URL fixa com --deploymentId

Organização dos arquivos .gs:

  • Code.gs — CONFIG global, doGet, onOpen, funções de inicialização e setup
  • api.gs — funções públicas chamadas pelo frontend (submitRequest, getOpenErros, etc.)
  • db.gs — operações de banco (getSpreadsheet_, ensureSheets_, deleteSolicitacao_, updateResposta_)
  • logic.gs — validações, cálculo de SLA, getUsuarioContexto_, PROFILE_PERM_DEFAULTS_
  • audit.gs — auditLog_, logDebug_, maskEmail_, sanitizePayloadForLog_

Otimizações de performance implementadas:

  • Memoização de SpreadsheetApp — getSpreadsheet_() abre a planilha apenas uma vez por execução GAS
  • Cache de sheets — ensureSheets_() cacheado por 5 min no CacheService.getScriptCache()
  • Cache de usuário — getUsuarioContexto_() cacheado por 5 min por e-mail
  • Cache de configurações — getConfigData() cacheado por 10 min
  • Batch de escrita — setValues() em vez de N chamadas setValue() individuais
  • Batch de exclusão — filter+rewrite em vez de N chamadas deleteRow()
  • Audit log otimizado (v265) — flushAuditLogs_() usa appendRow() para 1 entrada (1 API call) e setValues() apenas para lotes maiores, eliminando o getLastRow() desnecessário no caso mais frequente
  • Write direto no status (v265) — updateSolicitacaoStatusFast_() escreve diretamente sem ler o valor atual, economizando 1 API call por submitResponse
  • Edição de resposta sem re-calcular status (v265) — updateResposta_() não chama updateSolicitacaoStatusFast_(), pois o status da solicitação não muda em edições (erro já estava corrigido)

Resultado v265 (conta consumer gratuita — quota 90 min/dia): submitRequest ~3 API calls, insertResposta ~4 API calls, updateResposta ~3 API calls. Estimativa: 1.000 operações/dia ≈ 41 min (46% da quota).

SPREADSHEET_ID: armazenado exclusivamente no PropertiesService.getScriptProperties() — nunca em código versionado. Execute o menu "Configurar Propriedades (rodar 1x)" no Google Apps Script após o primeiro deploy ou ao trocar a planilha.

🔒 Segurança e Privacidade (LGPD)

🔐 Autenticação

Google OAuth 2.0 — apenas usuários cadastrados na aba Usuarios têm permissões. Acesso não cadastrado é tratado como ESPECTADOR.

⏱️ Rate Limiting

Máximo 15 chamadas de escrita por minuto por usuário via CacheService. Exceder retorna erro sem bloquear operações de leitura.

🏢 Controle por Setor

Usuários RESPOSTA veem apenas registros do seu setor. Registros sem setor são visíveis somente para ADMIN e CONFERENTE.

📋 Audit Log

Todo CREATE/UPDATE/DELETE registra: usuário, tabela, campo, valor anterior, novo valor e timestamp. Imutável via interface.

🛡️ LGPD — Proteção de Dados

E-mails são mascarados nos logs de debug (jo***@empresa.com). Dados sensíveis (nome, requisição, e-mail) são removidos dos payloads de log. DEBUG desativado automaticamente em PROD.

🔒 Concorrência

LockService bloqueia gravações simultâneas para evitar IDs duplicados e conflitos de dados.

🐛 Modo Debug (Desenvolvedores):
  • Ctrl+Shift+D — ativa/desativa o painel de debug flutuante em tempo real
  • Logs do servidor gravados em Logs_Debug na planilha (máx. 1000 linhas, mais antigas removidas automaticamente)
  • Dados sensíveis são mascarados nos logs mesmo com debug ativo
  • DEBUG é automático: ativado no ambiente DEV, desativado no PROD
